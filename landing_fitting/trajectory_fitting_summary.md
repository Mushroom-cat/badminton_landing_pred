# 轨迹拟合实现、实验结果与论文差异分析

## 1. 当前实现方式

当前实现位于 `landing_fitting/fit_trajectory.py`，用于读取 `landing_fitting/trajectory.txt`，按论文 MV-BMR 第 3.2.2 节给出的三条轨迹公式进行拟合，并根据最后一行真实落点计算误差。

### 数据读取

- `trajectory.txt` 的所有非空行都会被解析为 `x,y,z` 三维坐标。
- 最后一行作为真实落点标签 `label_xyz`。
- 最后一行之前的所有点作为观测轨迹点参与拟合。
- 当前文件解析结果：
  - 非空点数：112
  - 观测轨迹点：111
  - 真实落点标签：`(162.839000, -68.789000, 1.567910)`

当前观测轨迹首尾点为：

```text
first_observed: (1008.200000, 59.830300, 251.047000)
last_observed : (354.696000, -44.704400, 112.907000)
```

观测点的 `z` 值从 `251.047` 连续下降到 `112.907`，111 个观测点中相邻高度下降次数为 `110/110`。因此，这段数据表现为击球后处于下降阶段的一段飞行轨迹，最后一行是真实落点标签，不是观测序列的一部分。

### 时间轴设定

当前默认使用：

```text
fps = 300
landing_z = 0
```

脚本用帧序号构造毫秒时间轴：

```text
t_i = i / fps * 1000
```

因此，当前 111 个观测点覆盖的时间长度为：

```text
(111 - 1) / 300 * 1000 = 366.666667 ms
```

### 拟合公式

当前实现直接采用论文中的函数形式：

```text
x(t) = a1 * exp(b1 * t) + c1
y(t) = a2 * exp(-0.002 * t) + b2 * t + c2
z(t) = a3 * exp(b3 * t) - 0.79 * t + c3
```

其中：

- `t` 的单位按论文公式理解为毫秒。
- 坐标单位按论文描述和当前项目数据习惯理解为厘米。
- `-0.002` 和 `-0.79` 保持论文公式中的经验常数。

### 参数求解方式

当前脚本使用 `scipy.optimize.curve_fit` 对每个轴的全部参数进行非线性最小二乘拟合：

```text
x: fit a1, b1, c1
y: fit a2, b2, c2
z: fit a3, b3, c3
```

然后使用 `scipy.optimize.brentq` 求解：

```text
z(t) = landing_z = 0
```

得到预测落地时间，再带入 `x(t)`、`y(t)`、`z(t)` 得到预测落点。

### 运行方式

默认运行：

```bash
python landing_fitting/fit_trajectory.py
```

显式指定参数：

```bash
python landing_fitting/fit_trajectory.py --input landing_fitting/trajectory.txt --fps 300 --landing-z 0
```

主要结果：

## 3. 与论文方法的主要差异

当前实现是论文公式的“纯曲线拟合版”，论文原方法更接近“物理/经验公式 + 神经网络预测关键参数 + 最小二乘拟合”的混合方法。

### 3.1 参数 `c1/c2/c3` 的来源不同

论文中，`c1/c2/c3` 不是简单和 `a/b` 一起从当前点集全参数拟合出来的。论文描述为：

- 先用最新 20 个轨迹点做多项式拟合。
- 从这些点中计算当前状态特征，例如位置、速度、加速度等。
- 用神经网络模型预测 `c1/c2/c3`。
- 在 `c` 已知或由网络预测后，再用所有已检测点通过最小二乘拟合其余参数。

论文实验 Table 4 比较了 LSTM、CNN、MLP 对 `c1/c2/c3` 的预测效果，其中 MLP 的误差和推理时间表现最好。

当前实现没有训练或加载 MLP，而是直接对每个轴的 `a/b/c` 全部拟合：

```text
当前实现: observed points -> curve_fit -> a,b,c
论文方法: recent states -> MLP -> c, then least squares -> remaining parameters
```

这是当前实现和论文方法最核心的差异。

### 3.2 当前实现不是实时增量预测

论文方法服务于机器人实时击球控制：

- 相机持续检测羽毛球位置。
- 当检测点数超过 20 后，开始持续预测后续轨迹。
- 随着新点进入，预测不断更新。
- 预测结果用于选择机器人击球点，而不只是求地面落点。

当前实现是离线脚本：

- 一次性读取 `trajectory.txt`。
- 用前 111 个观测点统一拟合。
- 解出 `z=0` 的地面落点。
- 与最后一行真实落点计算误差。

因此当前实现更适合验证公式和单条轨迹拟合效果，不等同于论文中的在线击球点预测模块。

### 3.3 当前实现没有使用论文第一阶段 SEPNet

论文完整预测系统有两个阶段：

1. 击球初期使用 SEPNet，融合羽毛球轨迹和球拍 IMU 信息，提前预测机器人 hit zone。
2. 累积足够轨迹点后，使用曲线拟合和 MLP 预测完整轨迹。

当前脚本只实现第二阶段公式拟合的一部分，不包括：

- 球拍 IMU 数据。
- SEPNet。
- hit zone 分类或中心预测。
- 机器人平台控制。
- NMPC 击球控制。

### 3.4 当前实现没有论文中的数据清洗和训练流程

论文中用于训练网络的数据来自其实验系统采集的多组对打轨迹，并经过筛选和增强。论文提到：

- hit zone 预测部分收集了约 5000 个样本，经过过滤和增强。
- hit point prediction error 的统计中使用了约 450 条轨迹。
- `c1/c2/c3` 的 MLP 训练应当基于完整轨迹构造监督标签，再用早期或当前状态特征作为输入。

但论文没有公开完整的 `c1/c2/c3` 数据格式、MLP 网络结构和训练集文件。因此当前仓库在没有这些数据和模型的情况下，无法完全复现论文的 MLP 部分。

## 4. 如果要进一步靠近论文实现

要从当前版本升级到更接近论文的版本，需要补充以下内容：

1. 构建多条完整轨迹数据集。
2. 对每条完整轨迹离线拟合或优化出目标 `c1/c2/c3`，作为监督标签。
3. 对每条轨迹的前 20 个点或最新 20 个点做多项式拟合，提取位置、速度、加速度等输入特征。
4. 训练一个小型 MLP，输入当前状态特征，输出 `c1/c2/c3`。
5. 推理时先用 MLP 预测 `c`，再用所有已检测点拟合剩余参数。
6. 将预测从一次性落点计算改为随新点更新的实时轨迹预测。

如果目标只是对当前 `trajectory.txt` 做落点误差验证，当前实现已经足够清晰直接；如果目标是复现论文方法，则下一步重点应当是构造 `c1/c2/c3` 的训练数据和 MLP。
